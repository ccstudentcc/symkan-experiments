# Security Policy

## 适用范围

本仓库主要包含研究代码、实验脚本与结果分析文档，但相关安全问题仍应按照正式流程处理。当前重点关注以下风险：

- 凭证、密钥或令牌被硬编码到仓库。
- 结果导出、配置读取或脚本执行过程中存在明显的路径注入或命令注入风险。
- 外部数据下载、加载与缓存流程缺乏基本边界控制。
- 会导致用户本地环境、模型产物或实验数据受损的高风险行为。

## 漏洞报告

若发现安全问题，请不要直接在公开 issue 中披露细节。报告中建议至少包含以下内容：

1. 问题描述。
2. 影响范围。
3. 复现步骤或触发条件。
4. 风险等级的初步判断。
5. 可行的缓解思路（如有）。

若问题涉及密钥、令牌、个人数据或可直接利用的技术细节，应采用私下披露方式。

## 披露要求

在问题确认或修复完成之前，请避免以下行为：

- 公开 exploit 细节；
- 提交包含真实密钥的复现样例；
- 在 issue、pull request 或文档中粘贴敏感配置。

## 项目约束

- 不在代码、文档或样例中写入 API key、token、cookie 或私有路径。
- 凭证应通过环境变量传递，例如 `OPENAI_API_KEY`。
- 不因“便于复现”而提交未经确认可公开分发的本地数据快照。

## 响应原则

维护者会优先确认以下问题：

1. 该问题是否真实存在。
2. 其影响范围与破坏面是什么。
3. 是否存在更小、更清晰且兼容现有工作流的修复方式。

在风险得到确认后，修复过程将优先兼顾安全性与现有用户工作流的稳定性。
