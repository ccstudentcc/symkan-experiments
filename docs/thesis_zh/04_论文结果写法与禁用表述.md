## 1. 默认流水线的写法

推荐写法：

“当前工程版默认路径并不是经验性堆叠，而是由总体 rerun、模块消融和 LayerwiseFT 专题共同约束出来的折中方案。”

不推荐写法：

“本文默认流水线在所有指标上都优于其他设置。”

前一句与现有证据一致，后一句超出了当前材料能支撑的范围。

## 2. Stagewise 的写法

推荐写法：

“在当前实现中，Stagewise Train 更接近可符号化入口的必要准备，而不是普通调参开关。”

不推荐写法：

“Stagewise Train 带来显著精度提升，因此应无条件开启。”

现有材料更强地支持“可用性边界”，而不是“绝对精度收益”。

## 3. Input Compaction 的写法

推荐写法：

“Input Compaction 对应的是运行成本与公式数值一致性之间的显式权衡。”

不推荐写法：

“Input Compaction 在当前实验中全面改善了结果。”

因为现有报告已经说明，它会牺牲一部分公式验证侧表现。

## 4. LayerwiseFT 的写法

推荐写法：

“在当前 2 层 KAN 设定下，LayerwiseFT 更适合作为可选实验开关，而不是默认收益模块。”

不推荐写法：

“LayerwiseFT 无效”或“LayerwiseFT 已被彻底否定”。

当前证据只支持“默认不开更稳”，不支持彻底否定这一类技术路线。

## 5. ICBR backend 的写法

推荐写法：

“在 shared numeric、trace rhythm 与 shared symbolic-prep 对齐的前提下，ICBR 在 layered paired 切片上表现出更低的 `symbolic_core_seconds`，同时质量指标也有所改善。”

对 FAST_LIB 更适合写成：

“在更大候选库下，ICBR 保持了更强的核心符号阶段提速，而质量整体接近持平。”

不推荐把 layered 与 FAST_LIB 压成一句“ICBR 全面优于 baseline”，也不推荐把 `baseline_icbr_fulllib` 补写成 paired 结论。

## 6. 结果讨论里容易出现的误写

- 把 `formula_export_success_rate` 当成真实公式恢复成功率。
- 用 `symbolize_wall_time_s` 代替 `symbolic_core_seconds` 作为 backend 主速度指标。
- 把历史 rerun、模块消融和 paired compare 混成同一张大表后再下总判断。
- 用答辩 slide 里的摘要句替代原始 report 的边界说明。

## 7. 一个更稳妥的段落骨架

可以直接按下面的顺序组织实验段落：先交代当前工程版总体基线，再说明默认流水线由哪些模块职责约束而来，随后才进入 shared-state 对齐后的 backend compare，最后再补充 FAST_LIB 与 full-library 的边界。
