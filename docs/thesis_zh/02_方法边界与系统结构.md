## 1. 方法章应先写清什么

如果论文正文要围绕 `symkan-experiments` 写方法章，最重要的不是堆命令，而是先把四个边界说清楚：配置边界、阶段边界、比较边界、导出边界。

## 2. 配置边界

当前仓库把 `AppConfig` 作为统一配置对象。它的作用不是让代码更“整洁”，而是把 notebook、CLI 与脚本运行口径收拢到同一处，减少不同入口之间的默认值漂移。

写作时更稳妥的句子是：本文通过统一配置对象约束实验入口，使训练、符号化和结果导出的主要参数都能回指到同一套配置边界。

## 3. 阶段边界

现有文档已经把主流程拆成四段。

1. 数值训练与阶段筛选。
2. shared symbolic-prep。
3. backend-specific symbolic completion。
4. 数值验证与结构化导出。

这种拆分的意义在于：它把“模型是否可符号化”“两个后端是否公平比较”“结果能否回到结构化文件”分成了不同问题，避免整条流程被写成一个无法归因的黑箱。

## 4. 比较边界

当论文要比较 `baseline` 和 `baseline_icbr` 时，真正需要强调的是 shared numeric、trace rhythm 与 shared symbolic-prep 已经对齐。只有在这个前提成立时，后续差异才可以被收束为 backend-only compare。

对应的写法不应停留在“两个配置只差一个 backend”这种口头说明，而应明确引用现有 shared-check 文档或 report 中的对齐结论。

## 5. 导出边界

当前仓库把结果落到 `symkanbenchmark_runs.csv`、`kan_stage_logs.csv`、`symbolize_trace.csv`、`formula_validation.csv`、`metrics.json` 以及 compare 目录下的汇总表中。正文若要写实验结论，最好总能回指到这类结构化文件，而不是回指一段终端输出或 notebook 截图。

## 6. 方法章中适合保留的系统结构描述

可直接复用的简述如下：

本文方法在不改变参考 KAN 数值训练语义的前提下，将训练后符号化流程拆为统一配置、阶段化数值准备、共享符号预处理、后端特异的符号完成、数值验证与结构化导出几个环节。这样做的目的不是增加流程复杂度，而是让不同层级的实验结论拥有清晰的归因边界。
